# Implementation Plan: Printable Forms vs Receipt/Ticket

## Goal
Implement two distinct print workflows in this codebase:

1. **Generate Record Printable Form (manual, multi-record/report)**
2. **Print Receipt/Ticket (immediate, single transaction)**

while preserving current patterns, routes, and role permissions.

## Current System Findings (Repo-Specific)

- Immediate transaction receipt infrastructure already exists:
  - `backend/classes/ReceiptRepository.php`
  - `app/librarian/receipt.php`
  - `public/js/page-alerts.js` (opens print view/download after successful action)
- Receipt creation is already wired into transaction methods:
  - `LibrarianPortalRepository::checkInLoan()`
  - `LibrarianPortalRepository::checkoutLoan()`
  - `LibrarianPortalRepository::checkoutReadyReservation()`
  - `CirculationRepository::createBorrowerReservation()`
- Legacy receipt APIs are retired (`410`) and should remain unused:
  - `api/receipts/create.php`
  - `api/receipts/reprint.php`
- Report/list screens currently show data but do not provide dedicated printable-form endpoints:
  - `app/librarian/circulation.php`
  - `app/librarian/reservations.php`
  - `app/librarian/fines.php`
  - `app/admin/admin-fines.php`

## Target Design

### A) Generate Record Printable Form (manual)

Use a dedicated **print-view route** that renders document-style HTML for reports/lists.

- Trigger: user clicks **Generate Printable Form** button.
- Scope: multi-record and report data.
- Output: clean print layout, optional `auto_print=1` to run `window.print()`.

### B) Print Receipt/Ticket (immediate)

Keep current transaction-confirmation behavior and standardize naming/labels.

- Trigger: success of checkout/check-in/reservation action.
- Scope: single transaction receipt row (`transaction_receipts`).
- Output: receipt/ticket view from `librarian-receipt.php?receipt_id=...&auto_print=1`.

## Implementation Steps

### 1) Create report print-view endpoint

Add:

- `app/librarian/print-records.php` (new)
- `librarian-print-records.php` (root wrapper, same style as existing wrappers)

Responsibilities:

- Require authorization with `PermissionGate`.
- Parse and validate query params:
  - `type` (`circulation|reservations|fines|admin-fines`)
  - optional filters (`status`, `from`, `to`, etc. if added later)
  - `auto_print=1` optional
- Load data from existing repositories:
  - circulation: `LibrarianPortalRepository::getCirculationRows()`
  - reservations: `LibrarianPortalRepository::getReservationQueue()`
  - fines/admin-fines: `FineReporting::getMonthToDateReport()`
- Render a consistent printable document shell:
  - report title
  - generated timestamp
  - optional filter summary
  - tabular records
  - totals section when applicable
- Add print-only script:
  - if `auto_print=1`, call `window.print()` on load.

### 2) Add “Generate Printable Form” entry points

Update pages with a button/link to print-view route:

- `app/librarian/circulation.php`
- `app/librarian/reservations.php`
- `app/librarian/fines.php`
- `app/admin/admin-fines.php`

Button behavior:

- opens print form in new tab (`target="_blank" rel="noopener"`).
- URL format example:
  - `appPath('librarian-print-records.php', ['type' => 'circulation'])`

### 3) Keep and refine immediate receipt/ticket flow

No architecture replacement needed; refine labeling and consistency.

Adjust:

- `app/librarian/receipt.php`
  - map `transaction_type` to user-facing heading:
    - `checkout`, `checkin`, `reservation_checkout` -> `Receipt`
    - `reservation_create` -> `Reservation Ticket`
- (Optional) include a short document subtype label in metadata block.

Continue using existing alert integration:

- `app/librarian/circulation.php`
- `app/librarian/reservations.php`
- `public/js/page-alerts.js`

This preserves current behavior:

- successful action -> confirmation dialog -> print/download action.

### 4) Print styling standardization

Add a reusable print stylesheet for report pages (recommended):

- `public/css/print.css` (new)

Use in print-view route for:

- page margins and typography
- table header repeat/readability
- hide non-document UI in print mode
- page-break-safe sections

Keep existing page-specific styles intact for non-print pages.

### 5) Security and robustness checks

- Enforce role access in print-view endpoint via `PermissionGate`.
- Sanitize all incoming params and bound query values.
- Fail safely:
  - unknown `type` -> `400`
  - unavailable dataset -> readable fallback message
- Ensure no retired receipt API endpoint is reintroduced.

## Acceptance Criteria

1. Librarian can generate printable forms for:
   - active circulation records
   - reservation queue
   - fine report rows
2. Admin can generate printable form for admin fine report.
3. Successful checkout/check-in/reservation actions still open immediate receipt/ticket print flow.
4. Reservation creation is represented as a **ticket** label in print view.
5. Print forms are readable on paper/PDF and contain generation timestamp plus core record fields.

## Rollout Notes

- Implement in small, safe increments:
  1) print-view endpoint
  2) buttons on librarian pages
  3) admin button
  4) receipt/ticket label cleanup
- Keep old flows functioning during rollout; this is additive and backward-compatible.
