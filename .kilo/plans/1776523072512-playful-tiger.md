# Borrower UI/UX Consistency Plan

## Goal
Make the entire borrower experience feel like one coherent product by unifying navigation, page structure, styling, and interaction patterns across dashboard, catalog, reservations, history, and account settings.

## Current State (Findings)
- Borrower pages currently use **two different navigation paradigms**:
  - `app/user/index.php` uses a right-side dashboard sidebar (`.sidebar`, `.sidebar-nav`) and custom dashboard styling.
  - `app/user/catalog.php`, `app/user/reservations.php`, `app/user/history.php`, and `app/user/account.php` use a top `borrower-navbar`.
- Navigation markup is duplicated in 4 borrower pages, increasing drift risk and making future UI changes error-prone.
- UI styles are split between `public/css/borrower.css` and large per-page inline `<style>` blocks, causing inconsistency in spacing, cards, tables, forms, and visual hierarchy.
- `account.php` has extra layout artifacts (custom footer + back-link treatment) that do not match other borrower pages.
- Mobile behavior relies mostly on nav wrapping, not a deliberate compact navigation interaction.

## Recommended UX Direction
Adopt **one shared top borrower navbar** across all borrower pages, including dashboard, then standardize all pages on the existing `borrower-page` / `borrower-shell` content framework.

Rationale:
- Minimal cognitive switching for borrowers.
- Lowest implementation risk compared to converting all pages to dashboard sidebar layout.
- Enables a single reusable navigation component and simpler responsive behavior.

## Implementation Plan

### 1) Componentize Shared Borrower Navigation
1. Create a shared partial for borrower nav (example target: `app/user/partials/borrower-navbar.php`).
2. Pass `currentPage` and current user first name into the partial to render active link state and greeting consistently.
3. Replace repeated nav HTML in:
   - `app/user/catalog.php`
   - `app/user/reservations.php`
   - `app/user/history.php`
   - `app/user/account.php`
4. Refactor `app/user/index.php` logged-in view to consume the same shared nav component.

### 2) Unify Borrower Page Layout Contract
1. Define a common borrower layout pattern for all pages:
   - `<main class="borrower-page">`
   - `<div class="borrower-shell">`
   - `<header class="borrower-page-header">`
   - consistent alert slot and section spacing
2. Apply that pattern to dashboard (`app/user/index.php`) so it aligns with catalog/history/reservations/account structure.
3. Remove one-off footer/back-link patterns on borrower pages unless intentionally standardized across all borrower pages.

### 3) Consolidate Styling into `public/css/borrower.css`
1. Move borrower page inline styles from:
   - `app/user/index.php`
   - `app/user/catalog.php`
   - `app/user/reservations.php`
   - `app/user/history.php`
   - `app/user/account.php`
   into organized sections in `public/css/borrower.css`.
2. Establish shared style groups:
   - nav (desktop + mobile)
   - page shells/headers
   - cards/panels
   - alerts/empty states
   - tables/status badges
   - forms/settings cards
   - button hierarchy
3. Keep `main.css` as global/base only; borrower-specific visual rules live in `borrower.css`.

### 4) Improve Borrower UX Consistency
1. Standardize page heading tone and subtitle format (parallel phrasing and action-oriented guidance).
2. Normalize button semantics and hierarchy:
   - primary = main task per page
   - secondary = supporting action
   - danger = destructive action only
3. Improve mobile nav usability (single-row strategy + controlled overflow or compact toggle pattern).
4. Standardize table behavior:
   - consistent row height, typography, and badge system
   - predictable horizontal scrolling wrapper on narrow screens
5. Standardize empty/error/success states with concise user guidance.

### 5) Dashboard Harmonization
1. Keep dashboard metrics and quick actions, but restyle to match borrower design language (same radii, spacing scale, card depth, heading rhythm).
2. Align dashboard quick actions with navbar destinations for predictable navigation flow.
3. Ensure dashboard no longer feels like a separate admin-like surface from the other borrower pages.

### 6) Accessibility and Interaction Polish
1. Ensure keyboard-visible focus states on all nav links, buttons, and form fields.
2. Preserve `aria-current="page"` on active nav item from shared component logic.
3. Verify color contrast for accent buttons, muted text, badges, and alerts.
4. Confirm sticky nav does not hide anchored content and behaves correctly across breakpoints.

### 7) Verification and Regression Checks
1. Functional checks (borrower logged in):
   - Dashboard (`index.php`)
   - Catalog reserve flow (`catalog.php`)
   - Reservations cancel flow (`reservations.php`)
   - Loan renew flow (`history.php`)
   - Account profile/password flows (`account.php`)
2. Responsive checks: desktop, tablet, and mobile widths.
3. Run PHP syntax checks on touched files.
4. Validate that nav links, active states, and alert rendering remain correct after partial extraction.

## Acceptance Criteria
- Borrower pages use one consistent navigation system and visual language.
- Dashboard, catalog, reservations, history, and account feel like one coherent suite.
- Navbar is rendered from a single shared component with correct active states.
- Inline per-page style blocks are removed or minimized in favor of centralized borrower stylesheet rules.
- Mobile usability is improved with intentional nav behavior and readable content layouts.
- Core borrower actions (reserve, cancel, renew, update account) remain fully functional.
