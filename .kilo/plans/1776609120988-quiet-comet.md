# Auto-Receipt Generation & Printing Plan

## Goal
Add automatic receipt generation and print capability for every successful transaction in the canonical library transaction flow, with minimal risk to existing circulation behavior.

## Success Criteria
1. Every successful write transaction creates a durable receipt record in the same DB transaction (or clearly documented fallback) before commit.
2. UI can immediately print a receipt after success without extra manual data lookup.
3. Reprint is supported from stable receipt IDs.
4. No duplicate receipts are created on retries/page refreshes.
5. Existing tests still pass; targeted tests cover receipt creation paths.

## Current Architecture Findings (Where To Integrate)
- Canonical circulation writes happen in repository methods, not in `api/receipts/*`:
  - `backend/classes/LibrarianPortalRepository.php:533` `checkInLoan()`
  - `backend/classes/LibrarianPortalRepository.php:795` `checkoutLoan()`
  - `backend/classes/LibrarianPortalRepository.php:928` `checkoutReadyReservation()`
  - `backend/classes/LibrarianPortalRepository.php:1285` `updateReservationStatus()` (non-financial status changes)
  - `backend/classes/CirculationRepository.php:354` `createBorrowerReservation()`
  - `backend/classes/CirculationRepository.php:556` `cancelBorrowerReservation()`
  - `backend/classes/CirculationRepository.php:826` `renewBorrowerLoan()`
- Old receipt endpoints are explicitly retired:
  - `api/receipts/create.php:4`
  - `api/receipts/reprint.php:4`
- Existing receipt-like persistence exists only for fine reporting (`fine_collections.receipt_code`):
  - schema in `backend/config/schema.sql:87`
  - reporting in `backend/classes/FineReporting.php:35`
- Immediate success feedback is handled via alerts/flash in page handlers:
  - `app/librarian/circulation.php:41`
  - `app/librarian/reservations.php:41`
  - `public/js/page-alerts.js:22`

## Integration Strategy (Recommended)
- **Recommended placement**: generate receipt data in repository write methods right before `commit()` so receipt creation is atomically tied to business success.
- **Recommended print trigger**: pass `receipt_id`/`receipt_code` back to page handlers, then trigger a print action in the success alert callback (or redirect to printable receipt page).
- **Recommended model**: create a new generic `transaction_receipts` table for circulation + fine events; optionally map `fine_collections.receipt_code` into it for consistency.

## Data Model Plan
1. Create `transaction_receipts` table via migration (do not alter vendor files):
   - `id` (PK)
   - `receipt_code` (UNIQUE, human-readable)
   - `transaction_type` (enum/string: `checkout`, `checkin`, `reservation_checkout`, `reservation_create`, `reservation_cancel`, `renewal`, `fine_collection`, etc.)
   - `transaction_ref_id` (loan_id / reservation_id / fine_collection_id)
   - `borrower_user_id`, `actor_user_id`
   - `amount` (nullable decimal, for fines)
   - `payload_json` (snapshot line items/details used for print)
   - `created_at`
   - Optional unique idempotency key: (`transaction_type`, `transaction_ref_id`) to prevent duplicates.
2. Keep `fine_collections` as source for fine totals; optionally store cross-reference in `payload_json`.

## Service Layer Plan
1. Add `backend/classes/ReceiptRepository.php` (or similar) with:
   - `createForTransaction(PDO $db, array $data): array`
   - `findById(PDO $db, int $id): ?array`
   - `findByCode(PDO $db, string $code): ?array`
   - `findOrCreateByKey(...)` for idempotency.
2. Receipt code generation best practice:
   - Prefix by type/date (example: `RCPT-CKO-20260419-XXXXXX`)
   - Use cryptographically random suffix for collision resistance
   - Retry on unique-key collision.

## Write-Path Hook Plan (Atomic)
Implement receipt creation in these methods after core updates succeed, before `commit()`:
1. `LibrarianPortalRepository::checkoutLoan()` (`backend/classes/LibrarianPortalRepository.php:795`)
2. `LibrarianPortalRepository::checkInLoan()` (`backend/classes/LibrarianPortalRepository.php:533`)
3. `LibrarianPortalRepository::checkoutReadyReservation()` (`backend/classes/LibrarianPortalRepository.php:928`)
4. Optional policy-driven hooks for borrower self-service events:
   - `CirculationRepository::createBorrowerReservation()` (`backend/classes/CirculationRepository.php:354`)
   - `CirculationRepository::renewBorrowerLoan()` (`backend/classes/CirculationRepository.php:826`)
   - `CirculationRepository::cancelBorrowerReservation()` (`backend/classes/CirculationRepository.php:556`)

Each method returns added keys on success:
- `receipt_id`
- `receipt_code`
- `receipt_print_url` (or enough data for UI to build it)

## UI/Printing Plan
1. Add printable receipt route/page (server-rendered PHP):
   - Example target: `app/shared/receipt-print.php` or `app/librarian/receipt.php`
   - Read by `receipt_id`/`receipt_code`, validate permissions.
2. In librarian pages (`app/librarian/circulation.php`, `app/librarian/reservations.php`):
   - On successful transaction, include print metadata in `$page_alerts[]` (redirect or callback path).
   - Use current alert pipeline (`renderPageAlerts`) to open print view and call print.
3. Keep borrower pages without forced print unless explicitly required by product policy.

## Security & Reliability Best Practices
1. Enforce access control per receipt view via existing role gates (`PermissionGate`).
2. Never trust client-side amount/title values for receipt content; receipt payload must come from DB state at commit time.
3. Use DB transaction for business write + receipt insert together.
4. Add unique idempotency constraints to prevent duplicate receipts on retries.
5. Log receipt generation failures and roll back transaction if receipt is mandatory.

## Testing Plan
1. Unit/behavior tests for receipt generation function:
   - code format
   - uniqueness retry
   - payload completeness
2. Integration tests for each hooked transaction path:
   - success creates one receipt
   - failed transaction creates none
   - repeated submit does not duplicate with same idempotency key
3. UI smoke tests:
   - successful checkout/checkin shows alert and printable receipt opens
   - reprint by receipt code works for authorized users.
4. Full regression:
   - `php backend/tests/run.php`

## Rollout Sequence
1. Migration + ReceiptRepository
2. Hook `checkoutLoan()` and `checkInLoan()` first (highest operational value)
3. Add print page + alert integration
4. Hook `checkoutReadyReservation()`
5. Optional borrower-event receipts
6. Add reprint entry points from fines/circulation UIs

## Tradeoff Decision (Explicit)
- **Option A (Recommended): synchronous atomic receipts**
  - Pros: guaranteed consistency, no missing receipts
  - Cons: small added latency on write path
- **Option B: async receipt generation queue**
  - Pros: faster response time
  - Cons: eventual consistency, retry/reconciliation complexity

Given this codebase (vanilla PHP, server-rendered, transactional repository methods), Option A is the best fit.

## Implementation Notes
- Do not revive `api/receipts/create.php` and `api/receipts/reprint.php`; keep receipt flow attached to canonical transaction paths.
- Keep changes surgical: repository write paths, one new receipt persistence module, one printable view, minimal page-level wiring.
